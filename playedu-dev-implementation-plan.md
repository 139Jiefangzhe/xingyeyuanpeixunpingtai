# PlayEdu 二次开发实施计划与步骤

> **说明**：本计划按技术依赖关系组织，不绑定具体工期，各步骤可根据团队人力并行或串行调整。

---

## 步骤一：基础设施与环境基座搭建

**目标**：建立所有微服务共享的底层运行环境，后续服务均在此之上部署。

### 具体任务
1. 部署 **Nacos**（注册中心 + 配置中心），划分 `dev`、`test`、`prod` 命名空间，配置公共参数（数据库连接池、Redis、RocketMQ 地址）
2. 部署 **RocketMQ**（NameServer + Broker），创建核心 Topic：`course-learning`、`exam-result`、`live-start`、`file-transcode`、`point-change`、`train-remind`
3. 部署 **Redis Cluster**（缓存 + 会话 + 分布式锁），制定全局 Key 命名规范：`{svc}:{biz}:{id}`
4. 部署 **SkyWalking**（OAP + UI + Agent），确认 Java Agent 挂载方式（-javaagent 或 Sidecar）
5. 部署 **Sentinel 控制台**，预设网关流控规则模板（单 IP QPS、接口热点限流）
6. 部署 **MinIO**（对象存储），划分 Bucket：`video-bucket`（视频）、`doc-bucket`（课件文档）、`img-bucket`（图片/头像）、`cert-bucket`（证书模板）
7. 部署 **XXL-Job-Admin**（分布式任务调度），配置执行器注册策略
8. 搭建 **GitLab CI + Harbor** 流水线，确认镜像构建、推送、K8s RollingUpdate 流程跑通
9. 初始化 **Kubernetes 基础资源**：Namespace、Ingress Controller、ConfigMap/Secret 模板

### 产出物
- 可访问的 Nacos 控制台、RocketMQ Dashboard、SkyWalking UI、Sentinel 控制台
- 一套完整的 `k8s-manifests/base/` 基础部署 YAML（包含上述所有中间件）
- 公共配置模板 `shared-config.yml`（数据库、Redis、MQ 连接串）
- CI/CD Pipeline 示例跑通，能成功构建并部署一个 HelloWorld 服务到 K8s

## 步骤二：基座服务化迁移

**目标**：将 PlayEdu 单体后端拆分为独立的微服务骨架，前端保持复用，确保原有功能在拆分后 100% 可用。

### 具体任务
1. **源码梳理**：拉取 PlayEdu 开源后端代码，绘制原有模块依赖图，识别强耦合点（用户与课程的中间表、权限拦截器、全局异常处理）
2. **数据库拆分**：
   - 将现有单库拆分为 `db_user`（用户/部门/角色/权限）和 `db_course`（课程/章节/资源分类/学习记录）
   - 导出并清理交叉外键，服务间关联字段改为逻辑外键（仅存 ID，不做数据库级约束）
3. **服务拆分**：
   - 创建 `user-svc`：迁移 `sys_user`、`sys_dept`、`sys_role`、`sys_permission`、`sys_resource` 相关代码，保留原有 RBAC 接口契约
   - 创建 `course-svc`：迁移 `course`、`course_chapter`、`course_resource`、`course_category`、`learning_record` 相关代码，保留原有课程管理、学员学习进度 API
4. **服务间通信**：
   - 在 `course-svc` 中通过 **OpenFeign** 调用 `user-svc` 获取用户基础信息（`GET /api/users/{id}`）
   - 配置 Feign 拦截器，自动传递 `Trace-ID` 和 `X-User-Id`
5. **网关接入**：
   - Spring Cloud Gateway 添加路由规则：`/user-svc/**` → user-svc，`/course-svc/**` → course-svc
   - 网关层实现 JWT 统一校验，解析 Token 后向下游服务写入请求头 `X-User-Id`
6. **前端适配**：
   - 梳理 PlayEdu 原有 React 前端代码，确认 `edu-pc`（学员端）和 `edu-admin`（管理端）可直接复用
   - 将前端请求基地址改为 Gateway 地址，调整接口路径前缀匹配网关路由
   - 如需，引入前端代理配置解决本地开发跨域

### 产出物
- `user-svc` 和 `course-svc` 两个可独立运行的 Spring Boot 服务，通过 Gateway 统一对外暴露
- 原有 PlayEdu 功能（部门管理、用户导入、课程发布、视频学习、学习进度追踪）完整可用
- 数据库拆分脚本（含数据迁移脚本）
- 服务间 Feign 调用示例与网关路由配置 YAML
- 前端基座 `edu-pc` 和 `edu-admin` 能正常对接新网关

## 步骤三：通用能力沉淀

**目标**：建设所有业务服务共享的基础能力，避免各服务重复造轮子。

### 具体任务
1. **公共 Starter 封装**（`edu-common-starter`）：
   - 统一响应体 `Result<T>`（code / msg / data / version）
   - 全局异常处理器（业务异常、参数校验、权限拒绝）
   - MyBatis-Plus 自动配置 + 分页插件 + 多租户字段填充（内部使用保留字段，不启用隔离逻辑）
   - Redis 序列化配置（Jackson）、分布式锁工具、缓存注解封装
   - Knife4j（Swagger）自动配置，接口文档分组
   - Feign 调用拦截器（传递 Trace-ID / User-ID）
   - RocketMQ 生产者/消费者统一封装（含重试策略、死信队列）
2. **文件服务（file-svc）**：
   - 统一文件上传接口（支持分片断点续传，返回预签名 URL）
   - 视频转码任务下发（调用 MinIO + 异步转码服务，或对接腾讯云 VOD）
   - 文档转 PDF 预览（PDF.js 在线预览，后端做格式转换）
   - 图片缩略图/水印处理
3. **消息服务（msg-svc）**：
   - 通知模板管理（站内信/邮件/短信/企微 Bot）
   - 多渠道推送路由引擎（优先级：站内信 → 邮件 → 短信）
   - 消息记录与已读回执查询 API
   - 订阅 `course-learning`、`exam-result`、`live-start` 等 MQ Topic，自动触发通知
4. **前端组件库初始化**：
   - 基于 Ant Design 5 封装业务通用组件：StandardTable（分页/筛选/刷新）、UploadVideo（带进度）、RichTextEditor（富文本）、UserSelector（人员选择器）
   - Axios 实例封装：BaseURL 自动切换、Token 无感刷新、统一错误提示、请求防抖
   - Zustand Store 初始化：UserStore（登录态）、AppStore（全局配置）

### 产出物
- `edu-common-starter` 发布到内部 Maven 仓库，所有服务依赖此 Starter
- `file-svc` 和 `msg-svc` 可独立部署并对外提供 API
- 前端公共组件库目录初始化，包含 5+ 高频业务组件
- 前后端联调：文件上传 → 转码回调 → 消息通知 链路跑通

## 步骤四：核心业务域开发（考试 + 直播 + 培训）

**目标**：在基座之上，开发企学星最核心的三大业务模块。

### 子步骤 4.1：考试中心（exam-svc）

1. **题库管理**：设计 `edu_exam_question` 主表 + 子表（`question_option`、`question_blank`、`question_answer`），支持单选/多选/判断/填空/问答/组合题
2. **组卷引擎**：
   - 手动组卷：从题库勾选题目组成试卷
   - 智能组卷：按题型数量、难度分布、知识点覆盖、分数权重规则自动抽题（带 Redis 分布式锁防并发重复抽题）
3. **考试发布**：试卷 + 考生范围（部门/岗位/指定人）+ 时间窗口 + 时长限制 + 及格线 + 重考次数
4. **考试过程**：考生进入考试 → 题目缓存到 Redis（防篡改）→ 倒计时 → 自动保存答案 → 提交
5. **防作弊**：切屏次数监控、IP 校验、题目乱序、选项乱序、人脸识别预留接口
6. **评分与成绩**：客观题自动评分；主观题分配阅卷人、阅卷列表、成绩仲裁
7. **成绩分析**：最高分/最低分/平均分/及格率/分数段分布；个人成绩单 PDF 导出

### 子步骤 4.2：直播中心（live-svc）

1. **直播间管理**：创建直播间（推流地址/拉流地址/录制配置）、关联课程/培训项目、讲师权限分配
2. **流媒体对接**：对接腾讯云直播 SaaS（或阿里云直播），获取推流 URL 和播放 URL，支持 HLS/FLV 多线路
3. **直播互动**：弹幕消息（通过腾讯云 IM 或 WebSocket 信令）、问答墙、在线签到、在线人数统计
4. **录制回放**：直播结束自动触发录制转存 VOD，生成回放课程并关联到 `course-svc`
5. **直播统计**：观看人数峰值、人均观看时长、互动频次记录

### 子步骤 4.3：培训中心（train-svc）

1. **培训项目**：培训计划创建 → 添加任务节点（课程/考试/直播/作业）→ 设置节点完成规则（看完/看完+考试通过/积分达标）
2. **学习任务派发**：指定学员范围 → 下发任务 → 学员端展示任务列表与进度
3. **O2O 混合培训**：线上节点 + 线下签到节点组合，线下支持地理位置签到
4. **师徒制**：导师-学员绑定关系表、带教任务模板、导师评价、学员出师考核流程
5. **实操作业**：作业模板定义 → 学员文件提交 → 评审人打分 → 退回重做流程
6. **审批流**：轻量级状态机实现（培训申请审批、作业提交审批），预留扩展 Camunda 工作流引擎的接口
7. **新员工培训路径**：入职触发自动分配（对接 HR 系统或手动分配）、固定学习路径（Day1→Day7→Day30）

### 产出物
- `exam-svc`、`live-svc`、`train-svc` 三个服务上线，各自拥有完整的 DB 表结构、API 接口、Swagger 文档
- 前端新增考试页面（答题/倒计时/防切屏）、直播页面（播放器+弹幕+签到）、培训任务页面（任务列表+进度追踪）
- 考试 → 成绩 → 积分/通知链路、直播 → 回放 → 课程关联链路、培训 → 任务派发 → 完成度统计链路跑通

## 步骤五：运营与社交域开发

**目标**：构建学习社区、积分激励、人才发展三大运营体系，激活平台活跃度。

### 子步骤 5.1：学习社区（community-svc）

1. **内容模型**：帖子（图文/视频）、问答（悬赏积分）、文章（长文/知识沉淀）
2. **互动机制**：点赞、收藏、评论（嵌套回复）、转发、举报
3. **组织形态**：圈子（按部门/岗位/兴趣创建）、话题标签聚合
4. **内容审核**：敏感词过滤 + 人工审核后台；精华/置顶/加精操作
5. **与课程联动**：课程讨论区自动创建对应板块；学完课程引导学员发帖讨论

### 子步骤 5.2：积分商城（point-svc）

1. **积分规则引擎**：配置化规则表（完成课程 +10、考试通过 +20、每日签到 +5、社区发帖 +3），支持条件组合与优先级
2. **积分账户体系**：总积分、可用积分、冻结积分、过期策略；流水记录表（不可篡改设计）
3. **积分发放**：订阅 `course-learning`、`exam-result`、`community.event` 等 MQ Topic，异步发放积分
4. **排行榜**：Redis Sorted Set 实时计算个人周榜/月榜、部门榜
5. **积分商城**：商品管理（虚拟商品：课程解锁/证书加速/头衔；实体商品）、库存管理、兑换申请 → 审核 → 发放流程、抽奖/转盘活动

### 子步骤 5.3：人才发展（talent-svc）

1. **岗位能力模型**：岗位 → 能力维度 → 技能点 → 关联课程/考试/证书；胜任力雷达图数据接口
2. **学习地图**：新员工路径 / 晋升路径 / 转岗路径；节点解锁机制（前置条件校验：完成课程 A + 考试 B 通过）
3. **岗位认证**：认证标准配置（课程清单 + 考试分数线 + 工作年限）、申请入口、审核流程、电子证书生成（PDF 模板渲染）与验真接口
4. **人才画像**：聚合 `course-svc` 学习完成率、`exam-svc` 考试分数、`talent-svc` 技能覆盖度 → 生成员工能力档案 API

### 产出物
- `community-svc`、`point-svc`、`talent-svc` 三个服务上线
- 前端新增社区板块（论坛列表/发帖/问答）、积分商城（商品/兑换/排行榜）、个人学习地图（节点解锁/证书展示）
- 积分事件流（学习/考试/社区 → MQ → 积分发放 → 排行榜更新）完整闭环

## 步骤六：数据中台与统计分析

**目标**：构建离线 + 实时数据分析能力，为管理者提供决策支持。

### 具体任务
1. **数据同步链路**：部署 **Canal**（或 Debezium）监听 MySQL Binlog，将 `learning_record`、`exam_record`、`point_record` 等增量数据同步到 **ClickHouse** 或 **Elasticsearch**
2. **统计服务（stats-svc）**：
   - 离线报表：部门学习完成率、人均学习时长、课程热度 Top10、考试及格率趋势（从 ClickHouse 聚合）
   - 实时看板：当前在线人数、今日学习人数、课程实时播放量（从 Redis 定时落库 + 轻量聚合）
   - 人才画像聚合：跨服务调用 user/course/exam/talent 数据，生成管理侧员工能力档案
   - 报表导出：异步生成 Excel/PDF，提供下载链接（对接 file-svc）
3. **管理后台数据大屏**：Ant Design Pro 的 ProComponents + ECharts，展示部门学习完成率、考试分析、课程热度、人才梯队分布
4. **数据质量校验**：对比 MySQL 原始数据与 ClickHouse 聚合结果，确保同步链路不丢数

### 产出物
- `stats-svc` 上线，提供稳定的报表查询 API
- 管理后台新增「数据中心」模块，包含 5+ 核心报表页面和 1 个数据大屏
- Canal → Kafka → ClickHouse 数据链路稳定运行

## 步骤七：性能调优与安全加固

**目标**：确保平台在高并发场景下稳定，数据与接口安全合规。

### 具体任务

#### 性能调优
1. **慢 SQL 治理**：开启 MySQL 慢查询日志（> 1s），逐个优化；确保所有列表查询走索引，禁止 `SELECT *`
2. **缓存策略落地**：热点数据（用户信息、课程基础信息、权限菜单）接入 Caffeine 本地缓存 + Redis 二级缓存；缓存一致性策略：Cache Aside + 延迟双删
3. **Redis 热点 Key 治理**： bigkey 拆分；ZSet 排行榜定期截断（仅保留 Top1000）；Hot Key 本地缓存兜底
4. **接口压测**：使用 JMeter 或 K6 对核心接口（登录、课程列表、考试提交、直播进入）进行压测，定位瓶颈
5. **Sentinel 限流调参**：根据压测结果配置 QPS 阈值、慢调用降级规则、热点参数限流

#### 安全加固
1. **鉴权全覆盖**：所有业务接口必须通过 Gateway JWT 校验；内部 Feign 调用加入自定义 Token 防直连；管理接口二次校验管理员角色
2. **敏感数据脱敏**：手机号/身份证号/邮箱在日志和接口响应中脱敏；数据库密码字段 AES 加密存储
3. **操作审计日志**：在 `user-svc` 或独立审计模块记录所有管理操作（谁在什么时间做了什么修改，前后值对比），审计日志只读，禁止修改删除
4. **文件安全**：文件上传白名单校验（类型 + 大小）；视频 URL 使用预签名 Token（有效期 2 小时）防盗链；禁止前端直接访问对象存储
5. **考试安全**：切屏检测逻辑落地、考试期间禁止右键/开发者工具（前端层面）、答案传输加密

### 产出物
- 压测报告（并发数、吞吐量、平均响应时间、错误率）
- Sentinel 限流规则配置 YAML（可随服务一起发布）
- 安全审计日志表与查询后台
- 数据脱敏工具类（Jackson 序列化脱敏注解）

## 步骤八：文档沉淀与运维交接

**目标**：项目从开发阶段过渡到运维阶段，确保团队能独立维护与扩展。

### 具体任务
1. **API 文档**：所有服务通过 Knife4j 生成 Swagger 文档，导出为 Markdown/PDF，存放于内部 Wiki
2. **架构文档更新**：
   - 服务依赖拓扑图（谁调用了谁，通过 Feign 还是 MQ）
   - 数据库 ER 图（每个服务的核心表关系）
   - 缓存 Key 规范手册
   - MQ Topic 与事件格式说明
3. **部署手册**：
   - 开发环境一键启动脚本（Docker Compose）
   - 生产环境 K8s Helm Chart 说明
   - 配置中心（Nacos）关键配置项清单
   - 扩容操作手册（如何给某个服务加 Pod）
4. **故障排查 SOP**：
   - 服务注册不上 Nacos → 检查网络/IP/端口/命名空间
   - MQ 消费堆积 → 查看消费线程数/消费者实例/消息重试策略
   - 数据库连接池耗尽 → 检查慢 SQL/连接池配置/是否有长事务
   - Redis 内存告警 → 检查 bigkey/过期策略/是否被当作持久化存储
5. **培训与交接**：对内部运维团队做一次部署+排障培训，移交 Harbor 仓库权限、K8s 集群权限、数据库账号


> **并行策略提示**：
> - 步骤三可与步骤二后半段并行启动（common-starter 开发不依赖具体业务）
> - 步骤四中的三个服务可并行开发（exam / live / train 彼此无强依赖）
> - 步骤五中的社区与积分可并行，人才发展建议等学习地图数据明确后再启动
> - 步骤六的数据同步链路可与步骤五并行搭建

## 各步骤边界与接口契约

| 上游步骤 | 下游步骤 | 交付接口/数据 |
|---------|---------|-------------|
| 步骤二（基座） | 步骤三/四/五 | `user-svc` 用户查询 API、`course-svc` 课程基础数据 API |

| 步骤四/五（业务） | 步骤六（统计） | 各业务数据库 Binlog、关键聚合查询接口 |
| 步骤七（调优） | 所有已上线服务 | 限流规则、缓存规则、安全配置（热更新或重启生效） |
