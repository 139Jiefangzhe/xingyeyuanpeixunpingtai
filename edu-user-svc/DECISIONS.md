# edu-user-svc 设计决策记录

## 记录规范
- 每个决策按日期分组记录，包含：问题、决策、理由、反转条件
- 反转条件：什么情况下可以推翻这个决策

## 已确认决策

### 2026-05-05
- 问题：`edu-user-svc` 是否直接套用新的 `BaseEntity` 体系
- 决策：不套用，保留 legacy `users/departments/user_department/admin_roles` 表映射，作为只读桥接服务
- 理由：当前阶段共享 PlayEdu 现有数据库，旧表结构不符合微服务统一字段规范，强改会破坏兼容性
- 反转条件：当用户中心完成物理拆库和新表重建后，再切换到统一 `BaseEntity`

- 问题：本阶段是否开放用户和部门写接口
- 决策：仅开放查询接口，不提供新增、修改、删除能力
- 理由：单体后台仍是用户主写入口，先避免双写和权限模型重复实现
- 反转条件：当管理后台完全迁移到 `edu-user-svc` 后，再逐步开放写接口

- 问题：角色实体如何从单体迁移到用户中心
- 决策：使用 `Role` 作为桥接名称映射 legacy 表 `admin_roles`，当前只保留实体，不暴露角色 API
- 理由：先保留后续权限中心拆分需要的领域模型，同时避免提前复制单体权限逻辑
- 反转条件：当权限/角色能力正式拆分成独立聚合或服务时，再重建角色 API 与权限模型
