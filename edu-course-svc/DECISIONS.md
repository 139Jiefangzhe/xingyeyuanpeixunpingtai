# edu-course-svc 设计决策记录

## 记录规范
- 每个决策按日期分组记录，包含：问题、决策、理由、反转条件
- 反转条件：什么情况下可以推翻这个决策

## 已确认决策

### 2026-05-05
- 问题：课程中心首轮拆分时是否重构为全新课程模型
- 决策：`edu-course-svc` 先作为 legacy bridge，直接复用 `courses/course_chapters/course_hour/resource_course_category` 旧表结构
- 理由：当前目标是尽快打通 `train-svc -> course-svc` 真实链路，先保证查询和存在性校验可用，避免过早重构阻塞服务拆分
- 反转条件：当课程中心开始承接写操作、学习进度、视频点播等独立演进需求时，再统一做领域重构和分库迁移

- 问题：课程列表筛选是否立即引入新的 `type/status` 生命周期字段
- 决策：首版接口保持 legacy 兼容，只暴露 `titleLike/categoryId/isRequired/isShow` 等已落表字段
- 理由：旧版课程表没有统一的课程生命周期枚举，强行抽象会制造字段语义漂移
- 反转条件：当课程中心完成独立建模并补齐统一状态机后，再替换为新的状态枚举筛选

- 问题：课程适用部门关系是否在首轮一并迁移
- 决策：本轮不迁移 `course_department_user` 的服务能力，仅保留课程主表、章节、课时和分类关联
- 理由：当前高优先级是支撑培训任务的课程引用校验和基础只读查询，部门范围控制可后续补齐
- 反转条件：当培训项目需要按部门精确控制课程可见范围或课程授权时，再补充部门关系迁移

### 2026-05-06
- 问题：`edu-course-svc` 本地前后端联调如何脱离 MySQL/Nacos 独立启动
- 决策：新增 `local` profile，使用 H2 内存库、`schema-local.sql`/`data-local.sql` 初始化 3 门课程和章节课时种子数据，同时禁用 Nacos 和 Flyway
- 理由：培训任务配置页面只需要稳定的课程查询接口和存在性校验，本地种子数据足以支撑真实 UI 联调，且能显著降低启动成本
- 反转条件：当本地环境恢复共享 MySQL 或 Docker Compose 全栈启动时，可回退到正式数据库脚本与配置中心

- 问题：本地 Admin 前端跨端口访问课程服务时如何处理浏览器限制
- 决策：增加 `LocalCorsConfig`，仅在本地开发场景允许 `http://127.0.0.1:3001` 跨域访问
- 理由：当前多服务联调采用前端直连 `8081-8084` 的模式，需要服务端明确放行管理后台端口才能进行真实请求
- 反转条件：当本地开发统一收敛到 Gateway 代理或同域部署时，可移除服务级 CORS 放行

- 问题：课程管理页面需要 `type/coverUrl/课节 resourceUrl`，但 legacy 课程表没有对应结构化字段
- 决策：继续复用 `courses.extra` JSON，新增 `courseType`、`coverUrl`、`lessonResourceUrls` 三类元数据；课程主表仍保留 legacy 字段不做大改
- 理由：这样可以在不破坏旧表主体结构的前提下支撑课程管理前端真实创建/编辑，同时避免为本轮 MVP 额外引入新表或重构旧表
- 反转条件：当课程中心正式承接独立写模型并进入分库重构时，再把这些 JSON 元数据拆分到明确列或独立表

- 问题：课程分类展示需要可读名称，但当前迁移范围内只有分类 ID 关联表，没有分类主数据表
- 决策：`edu-course-svc` 暂时在服务内维护本地分类标签映射，并通过 `/api/v1/courses/categories` 暴露给前端
- 理由：培训和课程管理页面都需要稳定的分类下拉与标签展示，服务端统一映射比前端各自硬编码更容易收敛
- 反转条件：当分类中心或统一字典服务落地后，改为从真实分类服务读取并移除本地映射
