# edu-live-svc 设计决策记录

## 记录规范
- 每个决策按日期分组记录，包含：问题、决策、理由、反转条件
- 反转条件：什么情况下可以推翻这个决策

## 已确认决策

### 2026-05-05
- 问题：直播中心首轮是否直接接入腾讯云直播 API
- 决策：`edu-live-svc` 首版只管理直播间元数据，`pushUrl/playUrl/recordUrl` 作为已生成地址的存储字段保留
- 理由：当前目标是先完成服务骨架和培训任务的直播资源引用链路，云厂商签名和回调集成后续单独迭代
- 反转条件：当开始交付真实推流、拉流、录制与回放能力时，再接入腾讯云直播 SDK/API

- 问题：直播间状态如何流转
- 决策：首版使用单向状态机 `1-未开始 -> 2-直播中 -> 3-已结束`，不支持回退
- 理由：最小可用场景只需要表达预约、开播、结束三个阶段，单向流转可以降低状态复杂度
- 反转条件：当直播支持重播、回放重建或取消排期时，再补充更多状态和状态迁移规则

- 问题：直播间和课程的关系如何处理
- 决策：`courseId` 先作为逻辑外键保存，不在数据库层建立 FOREIGN KEY
- 理由：符合微服务架构下“逻辑外键、应用层校验”的约束，也便于后续接入 `course-svc`
- 反转条件：当课程中心对直播间关系提供稳定的批量校验/同步机制时，再在 Service 层增强校验和缓存

### 2026-05-06
- 问题：`edu-live-svc` 本地联调是否需要接入真实流媒体基础设施
- 决策：`local` profile 下继续只使用 H2 内存库和 `schema-local.sql`/`data-local.sql` 维护 2 个直播间种子数据，不引入腾讯云直播或外部推流依赖
- 理由：培训任务配置页面当前只需要拉取直播间列表并校验资源存在，直播 SaaS 不属于本轮真实联调的必需依赖
- 反转条件：当开始验证开播、停播、回放等真实直播链路时，再切回外部流媒体配置和云厂商接口

- 问题：本地多服务直连模式下如何允许 Admin 前端访问直播服务
- 决策：增加 `LocalCorsConfig`，允许 `http://127.0.0.1:3001` 在本地开发环境跨域访问直播接口
- 理由：当前前端通过 `request.ts` 按路径前缀直连各服务端口，如果没有显式 CORS 放行，浏览器会阻止培训任务页面加载直播资源
- 反转条件：当本地联调切回统一 Gateway 或反向代理时，可删除服务内的本地 CORS 配置
