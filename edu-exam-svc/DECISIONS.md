# edu-exam-svc 设计决策记录

## 记录规范
- 每个决策按日期分组记录，包含：问题、决策、理由、反转条件
- 反转条件：什么情况下可以推翻这个决策

## 已确认决策

### 2026-05-05
- 问题：`Question.options` 使用 JSON 字段还是独立表
- 决策：使用 JSON 字段存储选项
- 理由：选项数量少（2-5 个），当前不需要单独查询
- 反转条件：未来需要选项级统计分析时拆表

- 问题：本阶段是否实现 `QuestionOption` 独立表和对应 Mapper
- 决策：当前阶段仅实现 `Question` 实体与 `QuestionMapper`，不创建独立选项表
- 理由：本轮范围限定为服务骨架与题库题目基础建模，`options` JSON 已覆盖当前落库需求
- 反转条件：需要选项级统计、独立过滤或单项审计时新增 `edu_exam_question_option`

- 问题：`ExamPaper.knowledgeConfig` 采用什么存储结构
- 决策：使用 JSON 字符串存储组卷规则，内容承载题型数量、难度分布和知识点权重
- 理由：当前阶段仅需实体层和落库能力，JSON 能兼容手动组卷与随机抽题的后续扩展
- 反转条件：当组卷规则需要独立查询、版本化或复用时，拆分为独立配置表

- 问题：`publishPaper` 如何记录发布时间
- 决策：当前不新增 `publish_time` 字段，发布动作仅更新 `status=2`，并以 `updateTime` 作为最近发布时间痕迹
- 理由：现有 `db-schema.md` 试卷表没有 `publish_time` 字段，先保持与既有表结构一致
- 反转条件：当报表或审计需要独立发布时间时，为 `edu_exam_paper` 增加 `publish_time`

- 问题：`copyPaper` 的复制规则如何定义
- 决策：复制试卷时标题追加 `（副本）`，状态重置为草稿，并深拷贝所有题目关联记录
- 理由：副本需要可编辑且不能影响原试卷，同时保留原题目排序与分值配置
- 反转条件：当产品要求模板化复制或版本化命名时，改为独立副本命名策略

- 问题：`generatePaper` 的并发控制如何处理
- 决策：使用 Redisson 分布式锁，key 格式为 `lock:exam:generate:{bankId}:{ruleHash}`，获取失败立即返回“系统繁忙”
- 理由：同一题库与同一规则在并发场景下最容易重复抽题，锁住最小关键区可避免重复写入且不阻塞无关规则
- 反转条件：当后续改为异步组卷任务队列时，可改成队列幂等键而不是显式分布式锁

- 问题：`generatePaper` 的抽题算法采用哪一层随机
- 决策：先由 SQL 按 bank/type/difficulty/knowledgePoint 做过滤，再由 Java `Collections.shuffle` 随机并截取前 N 条
- 理由：SQL 负责收敛候选集，Java 层随机更易控制库存不足提示、去重逻辑和后续规则扩展
- 反转条件：当题库规模扩大到当前方案查询成本过高时，再改为数据库侧随机采样或预计算候选池

- 问题：考试评分采用实时评分还是异步批处理
- 决策：提交时立即计算成绩，采用实时评分
- 理由：用户体验要求立即看到成绩
- 反转条件：并发量超过 200 TPS 时改为异步

- 问题：`edu-common-starter` 尚未落地时，考试中心 Controller 响应格式如何统一
- 决策：在 `edu-exam-svc` 本地临时定义 `Result<T>`、`PageResult<T>` 和全局异常处理器，统一输出 `code/msg/data/version/timestamp`
- 理由：Controller 需要先稳定暴露接口契约，待公共 Starter 完成后再无缝迁移到共享实现
- 反转条件：当 `edu-common-starter` 提供统一响应模型和异常处理后，删除本地实现并切换到公共 Starter

- 问题：试卷详情接口如何展示试卷下题目
- 决策：`ExamPaperDetailResp` 返回按试卷顺序排列的 `List<QuestionResp>`，并将题目 `score` 视为试卷内实际分值
- 理由：前端查看试卷详情时更需要可直接渲染的题目数据，而不是原始关联表记录
- 反转条件：当客户端需要单独展示关联元数据（如独立排序对象或覆盖分值来源）时，补充专门的题目关联详情 DTO

- 问题：`exam_session` 在当前文档中没有正式字段定义时如何落地
- 决策：先实现最小运行态会话表，仅记录 `recordId/examId/paperId/userId/status/startTime/endTime/submitTime/lastActiveTime/ipAddress`，答案与评分结果仍只落在 `edu_exam_record`
- 理由：会话表本轮只负责超时自动交卷和活跃态管理，避免与考试记录表重复存储答案造成双写复杂度
- 反转条件：当正式 `exam_session` schema 发布，或需要断点续考/防作弊扩展字段时，再按正式模型扩表

- 问题：MVP 阶段没有独立考试实体和试卷快照表时，`exam_id/paper_id/dept_id` 如何处理
- 决策：`exam_id` 与 `paper_id` 暂同用已发布试卷 ID，`dept_id` 默认写 `0`，待上层在分配考试或接入用户上下文后补齐
- 理由：当前代码库只有试卷实体，没有单独考试发布聚合和快照表，但服务层需要先形成开考到评分的闭环
- 反转条件：当引入 `Exam` 发布实体、试卷快照表或用户上下文注入后，切换为真实 `exam_id`、快照 `paper_id` 和实际部门 ID

- 问题：多选题和主观题在自动评分阶段如何处理
- 决策：多选题采用“全对满分、漏选半分、错选 0 分”；单选/判断完全匹配得满分；填空/问答/组合题本轮标记为待阅卷，不参与自动得分
- 理由：这样可以保证客观题即时出分，同时不把主观题误判为已评分完成
- 反转条件：当后续接入主观题阅卷流或产品改动多选计分策略时，调整自动评分规则和状态流转

- 问题：存在主观题待阅卷时，考试记录状态如何流转
- 决策：提交后先置为 `2-已提交`，自动评分结束后若全部为客观题则更新为 `3-已评分`；若仍有主观题待阅卷则保持 `2`
- 理由：`3-已评分` 应只表示整份试卷的最终成绩已确定，避免把待阅卷记录误标为已评分
- 反转条件：当状态枚举扩展出“待人工阅卷”等更细状态时，改用新枚举承载中间态

- 问题：开始考试接口的用户身份从哪里获取
- 决策：`StartExamReq` 只传 `paperId`，`userId` 从 Gateway 注入的 `X-User-Id` Header 读取
- 理由：避免客户端伪造考生身份，并与微服务统一 Header 透传规范保持一致
- 反转条件：当统一鉴权上下文可直接注入登录用户对象时，可移除显式 Header 读取

- 问题：交卷接口是否需要立刻返回成绩结果
- 决策：`submitExam` Controller 在调用提交与自动评分后，立即返回 `ExamResultResp`
- 理由：交卷后展示总分和逐题明细是考试主链路，减少前端额外一次查分请求
- 反转条件：当评分改为异步流程或主观题成为主流程时，提交接口只返回受理结果，成绩改由独立接口轮询

- 问题：考试会话状态接口中的当前题号如何定义
- 决策：`currentQuestionIndex` 取试卷顺序中的第一道未作答题；若全部已答则返回最后一题序号，空试卷返回 `0`
- 理由：前端可直接恢复到下一道待答题，而不依赖本地缓存游标
- 反转条件：当会话表持久化前端游标或支持多端同步定位时，改为以持久化游标为准

- 问题：考试记录列表中的 `userName` 在当前阶段如何处理
- 决策：`ExamRecordListResp` 保留 `userName` 字段，但在未接入 `user-svc` 批量姓名查询前允许为空
- 理由：当前阶段禁止跨服务数据库访问，先稳定接口结构，后续通过服务调用补齐展示字段
- 反转条件：当 `user-svc` 提供批量用户信息查询接口后，再填充真实姓名

- 问题：接入 `user-svc` 后 `ExamRecord.deptId` 如何落库
- 决策：`startExam` 先通过 `user-svc` 校验用户存在，再取 `deptIds` 的第一个部门写入 `deptId`；缺失时回落 `0`
- 理由：当前考试记录表只保留单值 `dept_id`，先打通真实用户链路并满足基础统计场景
- 反转条件：当用户中心提供显式主部门字段或考试记录支持多部门关联时，改为写入主部门或独立关联表

- 问题：本地联调环境在 Nacos、MySQL、Redis 未就绪时如何启动考试中心
- 决策：提供 `local` profile，使用 H2 内存库启动，禁用 Nacos 配置/注册、Flyway、Redisson 与共享 Redis 配置，只保留最小 Web/API 能力
- 理由：当前阶段优先验证前后端字段和交互，不应被注册中心与基础设施缺失阻塞
- 反转条件：当本地 DevOps 基础设施稳定提供 Nacos、MySQL、Redis 后，回到统一外置配置和真实依赖启动

- 问题：考试中心枚举接口的返回结构与前端离线态如何统一
- 决策：`/api/v1/enums` 返回权威对象映射结构（如 `QuestionType -> {"1":"单选题"}`），前端允许通过 key alias 与本地 fallback 常量兼容 mock/离线模式
- 理由：对象映射更贴近字典接口表达，前端 alias/fallback 则保证旧 key 和断网联调都不阻塞
- 反转条件：当全站统一枚举中心落地且前后端 key 规范完全收敛后，删除 alias 和本地 fallback，仅保留统一契约

- 问题：培训项目详情需要真实考试通过率时，`exam-svc` 以什么粒度提供统计
- 决策：新增 `GET /api/v1/exam-records/papers/{paperId}/stats`，按试卷聚合考试记录，并以“每个用户取最近一条记录”作为统计口径
- 理由：培训中心按试卷关联考试任务，不应强行复用通用分页接口；按用户取最近一次记录可避免重考场景下重复计数
- 反转条件：当考试产品明确支持多次考试取最高分、第一次成绩或指定场次统计时，改为可配置的统计口径
